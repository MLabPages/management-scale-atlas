## 1. まず読む要約

このサイトは、経営学・マーケティング研究で使う構成概念と測定尺度を、原典確認だけでなく「実際の調査でどの版を採用するか」という観点から検索・比較する静的Webアプリである。

最重要方針は次のとおり。

1. 原版だけでなく、後続研究で実際に使われた短縮版・翻訳版・文脈調整版を選択肢として示す。
2. 複数概念を同時に測る研究で使いやすい3・4項目版を積極的に収録する。
3. 「原著論文の引用数」と「その尺度を実際に使用した研究数」を混同しない。
4. 日本語版を「厳密な検証済み」だけに限定せず、翻訳研究・使用例・関連版を根拠の段階別に表示する。
5. 原版、短縮版、翻訳版、研究内での項目削除版を同一物として扱わない。
6. 尺度項目本文は、公開・転載できる根拠が確認できた場合だけ掲載する。

## 3. ユーザーが求めているサイト像

原典だけを並べた辞典ではなく、研究者が次を判断できるサイトにする。

- 自分の研究文脈に近い使用研究はどれか
- 原版と短縮版のどちらが適切か
- 3・4項目で使われた実績があるか
- 日本語で使われたことがあるか
- 翻訳・逆翻訳や妥当性確認がどこまで行われたか
- 回答者負担を抑えつつ、内容妥当性を保てるか
- どの論文を採用根拠として引用すべきか

短い尺度を無条件に推奨してはいけない。項目数、因子構造、信頼性、AVE、内容範囲、測定不変性、翻訳手続、対象との近さを併記し、選択肢を提示する。

## 4. データ登録の重要ルール

### 4.1 原版レコード

最低限、次を一次資料または公式資料で確認する。

- 尺度名、著者、年、原典タイトル、掲載誌、DOI
- 原版項目数、下位次元、回答形式
- 対象者
- 版の種類
- 利用・転載条件

確認できない値を推測で埋めない。「本文で明示確認できず」など、未確認であることを値に残す。

### 4.2 `usageStudies`

論文が原典を引用しているだけでは登録しない。本文、方法、表、付録などから、その尺度または明示された調整版を実際に使用したことを確認する。

可能な限り次を記録する。

- `title`, `authors`, `year`
- `context`, `sample`
- `itemCount`, `responseFormat`, `language`
- `adaptation`: 翻訳、逆翻訳、文脈調整、項目削除など
- `result`: α、ω、CR、AVE、因子負荷、適合度など
- `doi`, `url`

研究内で削除された項目数と、最終分析に使われた項目数を区別する。原版と完全に同一でない場合は、`adaptation`で明記する。

### 4.3 `usageEvidence`

登録するのは、系統的レビュー等で「尺度を実際に使った」と判定された研究数である。

- Google ScholarやCrossrefの被引用数を入れない。
- レビュー対象期間、領域、母数を`scope`に書く。
- 全世界の累計と表現しない。
- 未集計は利用研究ゼロを意味しない。

### 4.4 日本語情報

`japaneseVersionStatus`は次を使い分ける。

- `validated`: 心理測定学的な検証済み
- `linguistic-validated`: 翻訳・意味的等価性を確認
- `usage-example`: 日本語訳・改変版の使用例
- `translation-study`: 翻訳、因子構造等の検討
- `related-version`: 同じ概念の関連版に日本語根拠あり
- `original-japanese`: 日本語で独自開発
- `unconfirmed`: 今回の確認範囲では未確認

「日本語で同じ概念を扱った論文」と「その尺度を日本語で使った論文」を混同しない。関連版を、対象尺度そのものの日本語版と表示しない。

### 4.5 尺度項目本文

次のいずれかを確認できるまで本文を掲載しない。

- 明確なオープンライセンス
- パブリックドメイン
- 著者・権利者・公式配布元による転載許可
- サイト掲載を許す利用規約

不明な場合は`usagePermission: "unknown"`、`itemPublicationStatus: "not-published"`を維持し、原典・公式入手先へのリンクだけを示す。

## 5. 今後のロードマップ

### P0: データ品質を自動検証する

低コストモデルでも安全に更新できるよう、最優先で検証スクリプトを追加する。

- 概念IDと尺度`conceptId`の参照整合性
- IDの重複
- DOI形式とURL形式
- 原版項目数と使用研究項目数の正の整数チェック
- `usageStudies`の必須項目
- 日本語ステータスの許容値
- 親尺度IDの存在
- 同一尺度内のDOI・研究タイトル重複
- `data.js`内のオブジェクトキー重複を検知する仕組み
- 集計値を出す`node`スクリプト

推奨成果物は`verify-data.mjs`。成功時に概念数、尺度数、使用研究数、短縮版数、日本語区分を表示し、異常時は終了コード1にする。

### P1: 実用性の高い尺度データを拡充する

優先順位は「利用頻度が高い」「3・4項目版が存在する」「日本語研究がある」「複数概念調査で使われやすい」で決める。

優先候補:

1. Brand Experience Scale
2. SERVQUAL
3. Brand Personality Scale
4. Consumer Brand Engagement Scale
5. Brand Trust Scale
6. Consumer Perceived Value / PERVAL
7. Entrepreneurial Orientation
8. Psychological Empowerment Scale
9. LMX-7
10. Job Crafting Scale
11. Maslach Burnout Inventory–General Survey
12. CETSCALE

各尺度について、最低2件を目標に異なる文脈の使用研究を登録する。3・4項目版がある場合は優先し、原版と同一か、抜粋か、再検証された短縮版かを区別する。

### P1: 日本語根拠を増やす

未確認19尺度を一括で「なし」と判断せず、1尺度ずつ調べる。

優先順:

1. 日本語版の開発・検証論文
2. 翻訳・逆翻訳手続を報告した研究
3. 日本語での明確な使用研究
4. 学位論文・紀要などの使用例
5. 関連版のみ存在する場合は別版として登録

### P1: 採用判断を支援する

- 使用研究を項目数、言語、対象、年で絞り込む
- 原版と選択中の後続版の差分を明示する
- 「短いが内容範囲が狭い」「翻訳済みだが妥当性未検証」などの注意を表示する
- 採用根拠文献をBibTeXまたはRISで出力する
- 同一概念内の尺度候補を比較しやすくする

単純な総合点やランキングを付ける場合は、評価基準と重みを公開する。根拠の異なる尺度をブラックボックスの点数だけで順位付けしない。

### P2: 尺度項目と利用条件

- 掲載可能な尺度だけ、項目本文・回答選択肢・逆転項目・得点化を登録
- 利用条件、出典、ライセンス、確認日を表示
- 掲載不可・不明の場合は、公式入手先への導線を改善

### P2: 派生関係の可視化

- 原版 → 短縮版
- 原版 → 翻訳版
- 原版 → 文脈調整版
- 類似概念・代替尺度

最初はテキストと表で実装し、データ構造が安定してから図にする。

### P2: 変更履歴と運用

- 尺度単位の`verifiedAt`と確認者
- データ変更履歴
- 根拠追加・訂正のIssueテンプレート
- 出典切れリンクの定期確認
- データ版番号とサイト機能版番号の整理

## 6. 次のモデルが最初に行う推奨タスク

大きな機能追加から始めず、次の順が安全である。

1. `verify-data.mjs`を作成する。
2. 現在の30概念・41尺度・27使用研究で検証を通す。
3. Brand Experience Scaleについて、原版12項目を実際に使った研究と短縮・項目削除版を各1件追加する。
4. 日本語での使用研究または翻訳根拠を1件探す。
5. 構文・データ検証後にコミット・公開する。

一度に多数尺度を追加するより、1～3尺度ずつ根拠を確認して公開する方が誤登録を発見しやすい。

## 7. 作業開始・検証・公開手順

### 作業開始

`C:\myproject\AGENTS.md`を読み、対象フォルダで必ず実行する。

```powershell
git pull
git status --short
git diff --stat
```

`git pull`は、GitHubなどの最新変更を手元へ取り込む操作。Claude Codeと同じフォルダを共有しているため省略しない。

### 最低限の検証

```powershell
node --check app.js
node --check data.js
git diff --check
```

データ検証スクリプト追加後は、必ずそれも実行する。

### 公開

```powershell
git add <変更したファイル>
git commit -m "変更内容を表す短い英語メッセージ"
git push
```

- `git commit`: 変更を履歴として保存する操作
- `git push`: 保存した履歴をGitHubへ送る操作

その後、GitHub Pagesの`pages-build-deployment`成功を確認する。HTTP 200だけでなく、公開された`data.js`や`app.js`に新しい版番号・尺度名・機能名が含まれることをコマンドで確認する。

## 8. このスレッド固有の運用制約

ブラウザ起動・ブラウザ操作がCodex再起動の一因である可能性があるため、このスレッドではブラウザを開かない。

使用する検証手段:

- `node --check`
- データ検証スクリプト
- PowerShellの`Invoke-WebRequest`
- `gh run list` / `gh run watch`
- 公開中のJS・CSS内のマーカー確認

画面クリック、保存復元、レスポンシブ表示など、ブラウザが必要な検証は「未実施」と明記する。実施していない画面テストを成功と報告しない。

## 9. 既知の注意点

- 静的サイトなので、データ量増加に伴い`data.js`が大きくなる。
- `data.js`はJavaScriptオブジェクトであり、同じオブジェクト内のキー重複が構文エラーにならず、後の値で上書きされる。自動検知が必要。
- ローカル保存はブラウザの`localStorage`依存で、端末間同期はない。
- 利用研究27件は全尺度を均等にカバーしていない。
- 利用研究数が未集計の尺度が多い。
- 日本語情報未確認が19尺度ある。
- 尺度項目本文は未掲載。
- 公開ページの完全なクリック操作テストは、直近のCodex再起動により未完了。構文、データ整合性、GitHub Pages、公開ファイルのマーカーは確認済み。

## 10. 完了条件

各更新は、次を満たして完了とする。

1. 根拠資料から項目数・対象・回答形式・調整方法を確認した。
2. 原版、短縮版、翻訳版、研究内削除版を区別した。
3. 引用数を尺度使用数として登録していない。
4. 日本語根拠の段階を正しく分類した。
5. 構文とデータ検証が通った。
6. 関係ない既存変更を上書きしていない。
7. GitHub Pagesの公開処理が成功した。
8. 公開ファイルに変更が反映されたことを確認した。
9. ブラウザ未使用の場合、画面操作テスト未実施を明記した。
