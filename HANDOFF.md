## 1. まず読む要約

このサイトは、経営学・マーケティング研究で使う構成概念と測定尺度を、原典確認だけでなく「実際の調査でどの版を採用するか」という観点から検索・比較する静的Webアプリである。

最重要方針は次のとおり。

1. 原版だけでなく、後続研究で実際に使われた短縮版・翻訳版・文脈調整版を選択肢として示す。
2. 複数概念を同時に測る研究で使いやすい3・4項目版を積極的に収録する。
3. 「原著論文の引用数」と「その尺度を実際に使用した研究数」を混同しない。
4. 日本語版を「厳密な検証済み」だけに限定せず、翻訳研究・使用例・関連版を根拠の段階別に表示する。
5. 原版、短縮版、翻訳版、研究内での項目削除版を同一物として扱わない。
6. 尺度項目本文は、公開・転載できる根拠が確認できた場合だけ掲載する。

## 2. 現在の実装状況

### データ

- 34概念
- 57尺度

- 登録版そのものが3・4項目: 11尺度
- 個別使用研究で3・4項目版を確認済み: 13尺度
- 日本語情報の内訳
  - 検証済み: 10尺度
  - 言語的妥当性確認: 1尺度
  - 日本語使用例: 6尺度
  - 日本語で独自開発: 5尺度
  - 翻訳・因子構造等の検討: 3尺度
  - 関連版の日本語根拠: 8尺度
  - 未確認: 24尺度

直近の経験研究拡充では、顧客経験のEXQ 19項目版とオムニチャネルCX Scale 18項目版、牧野・三好・瀨良（2026）の消費者の身体的反応6項目版および自動性3項目版を追加した。自動性3項目は6項目版からの単純な短縮ではなく、同じ項目生成過程から得られた別モデルとして扱う。知覚価値では、高橋（2018）の日本語で独自開発された6次元26項目の顧客価値尺度を追加し、PERVAL 19項目版と、日本語研究における3次元の選択・文脈調整例を別物として表示する。さらにWalshら（2014）が原データと英米3研究で再検証したPERVAL-Short 12項目版・8項目版、Sirdeshmukhら（2002）の総合知覚価値4項目版を追加した。総合4項目版にはクルーズでの4項目使用とAIサービス研究の3項目使用を登録した。消費者ブランド・エンゲージメントでは、Hollebeekらの10項目原版に観光文脈の再検証を追加し、髙橋（2022）の日本語12項目関連版を独立した選択肢として追加した。

続いて、Chaudhuri & Holbrook（2001）のブランド信頼4項目、ブランド感情3項目、ブランド・ロイヤルティ4項目を追加した。ブランド・ロイヤルティは購買2項目と態度2項目を区別する。ブランド信頼4項目は、Delgado-BallesterらのBrand reliability／Brand intentionsを分ける8項目BTSとは別尺度として扱う。

SERVQUALには、Cronin & Taylor（1992）による同じ22属性のSERVPERFとの比較、Babakus & Mangold（1992）の病院向け15項目適応版、松尾・奥瀬・プラート（2001）の日本語ホテル文脈における最終26項目版を個別使用研究として追加した。日本語研究は22項目に有形性5項目を加えた知覚のみ27項目から、共通性の低い1項目を除外して6因子26項目を分析したものとして、標準日本語版ではなく文脈適応版に分類する。EXQにはKlaus & Maklan（2013）の英国4サービス・計800名による19項目の後続検証を追加した。

ブランド・パーソナリティでは、Aaker原版42項目を9レストランで直接使用した研究を追加した。さらに、日本の全国郵送パネル1,495名から開発された5次元・12ファセット・36項目のJapanese Brand Personality Scaleと、Geuensら（2009）の5次元12項目尺度を独立登録した。日本版36項目は原版の翻訳・短縮ではなく、Peacefulnessを含む文化別再構成版である。Geuens版12項目もAaker原版からの項目削除版ではなく、ブランドに適用可能な人間の性格特性に構成概念を限定して新規開発された尺度であり、電気自動車文脈の使用研究を追加した。

Consumer Brand Engagementでは、HollebeekらのSNS向け10項目原版に、アラビア語へ逆翻訳してヨルダンのFacebook利用者500名へ適用した研究を追加した。さらに、Razmus（2021）がポーランド語の質的調査から新規開発したSNS外向け6項目版を独立登録した。6項目版は認知・感情・行動を各2項目で保持し、417名で精選、339名で因子構造、151名で再検査信頼性、98名で実験的妥当性を検討している。Hollebeek原版の研究内削除版ではない。原著がCC BYで項目を公開している尺度には、詳細画面から項目掲載ページへ移動できる表示も追加した。

Brand Trustでは、単一次元4項目版と、Brand reliability・Brand intentionsを各4項目で測るBTS 8項目版を別尺度として比較できるようにした。BTSにはシャンプー・ビール各140名での製品カテゴリー間測定不変性、オーストラリアのシャンプー利用者154名による2因子追試、米国大学生221名・75ファッションブランドでの使用研究を追加した。8項目BTSの日本語使用・翻訳検証は確認できず、4項目版の日本語双方向翻訳例を転用しないよう明記した。

## 3. ユーザーが求めているサイト像

原典だけを並べた辞典ではなく、研究者が次を判断できるサイトにする。

- 自分の研究文脈に近い使用研究はどれか
- 原版と短縮版のどちらが適切か
- 3・4項目で使われた実績があるか
- 日本語で使われたことがあるか
- 翻訳・逆翻訳や妥当性確認がどこまで行われたか
- 回答者負担を抑えつつ、内容妥当性を保てるか
- どの論文を採用根拠として引用すべきか

短い尺度を無条件に推奨してはいけない。項目数、因子構造、信頼性、AVE、内容範囲、測定不変性、翻訳手続、対象との近さを併記し、選択肢を提示する。

## 4. データ登録の重要ルール

### 4.1 原版レコード

最低限、次を一次資料または公式資料で確認する。

- 尺度名、著者、年、原典タイトル、掲載誌、DOI
- 原版項目数、下位次元、回答形式
- 対象者
- 版の種類
- 利用・転載条件

確認できない値を推測で埋めない。「本文で明示確認できず」など、未確認であることを値に残す。

### 4.2 `usageStudies`

論文が原典を引用しているだけでは登録しない。本文、方法、表、付録などから、その尺度または明示された調整版を実際に使用したことを確認する。

可能な限り次を記録する。

- `title`, `authors`, `year`
- `context`, `sample`
- `itemCount`, `responseFormat`, `language`
- `adaptation`: 翻訳、逆翻訳、文脈調整、項目削除など
- `result`: α、ω、CR、AVE、因子負荷、適合度など
- `doi`, `url`

研究内で削除された項目数と、最終分析に使われた項目数を区別する。原版と完全に同一でない場合は、`adaptation`で明記する。

### 4.3 `usageEvidence`

登録するのは、系統的レビュー等で「尺度を実際に使った」と判定された研究数である。

- Google ScholarやCrossrefの被引用数を入れない。
- レビュー対象期間、領域、母数を`scope`に書く。
- 全世界の累計と表現しない。
- 未集計は利用研究ゼロを意味しない。

### 4.4 日本語情報

`japaneseVersionStatus`は次を使い分ける。

- `validated`: 心理測定学的な検証済み
- `linguistic-validated`: 翻訳・意味的等価性を確認
- `usage-example`: 日本語訳・改変版の使用例
- `translation-study`: 翻訳、因子構造等の検討
- `related-version`: 同じ概念の関連版に日本語根拠あり
- `original-japanese`: 日本語で独自開発
- `unconfirmed`: 今回の確認範囲では未確認

「日本語で同じ概念を扱った論文」と「その尺度を日本語で使った論文」を混同しない。関連版を、対象尺度そのものの日本語版と表示しない。

### 4.5 尺度項目本文

次のいずれかを確認できるまで本文を掲載しない。

- 明確なオープンライセンス
- パブリックドメイン
- 著者・権利者・公式配布元による転載許可
- サイト掲載を許す利用規約

不明な場合は`usagePermission: "unknown"`、`itemPublicationStatus: "not-published"`を維持し、原典・公式入手先へのリンクだけを示す。

## 5. 今後のロードマップ

### P0: データ品質を自動検証する（v0.15.0で実装済み）

`verify-data.mjs`で、低コストモデルでも安全に更新できるよう次を検証できる。

- 概念IDと尺度`conceptId`の参照整合性
- IDの重複
- DOI形式とURL形式
- 原版項目数と使用研究項目数の正の整数チェック
- `usageStudies`の必須項目
- 日本語ステータスの許容値
- 親尺度IDの存在
- 同一尺度内のDOI・研究タイトル重複
- `data.js`内のオブジェクトキー重複を検知する仕組み
- 集計値を出す`node`スクリプト

成功時には概念数、尺度数、使用研究数、短縮版数、日本語区分を表示し、異常時は終了コード1にする。`node verify-data.mjs --self-test`では検証器の主要な異常検出も確認できる。

検証対象はリポジトリ内の`data.js`に固定する。`data.js`はJavaScriptとして評価するため、未確認の外部ファイルや外部PRを安全に検査する用途には使わない。その運用やCIを導入する場合は、実行を伴わないJavaScriptパーサー方式へ切り替える。

### P1: 実用性の高い尺度データを拡充する

優先順位は「利用頻度が高い」「3・4項目版が存在する」「日本語研究がある」「複数概念調査で使われやすい」で決める。

マーケティング領域は特に手厚くする。直近はブランド経験・顧客経験・消費者の身体的反応を同じ経験研究クラスターとして比較できるようにし、日本語の顧客価値尺度、PERVALの12・8項目短縮版、総合知覚価値4項目版、カスタマー・ブランド・エンゲージメント関連版も追加した。次は各尺度の個別使用研究と日本語根拠を増やしつつ、顧客ロイヤルティ、顧客満足、ブランド・エクイティの短い実用版を優先する。

優先候補:

1. Consumer Perceived Value / PERVAL
2. Entrepreneurial Orientation
3. Psychological Empowerment Scale
4. LMX-7
5. Job Crafting Scale
6. Maslach Burnout Inventory–General Survey
7. CETSCALE

各尺度について、最低2件を目標に異なる文脈の使用研究を登録する。3・4項目版がある場合は優先し、原版と同一か、抜粋か、再検証された短縮版かを区別する。

### P1: 日本語根拠を増やす

未確認24尺度を一括で「なし」と判断せず、1尺度ずつ調べる。

優先順:

1. 日本語版の開発・検証論文
2. 翻訳・逆翻訳手続を報告した研究
3. 日本語での明確な使用研究
4. 学位論文・紀要などの使用例
5. 関連版のみ存在する場合は別版として登録

### P1: 採用判断を支援する

- 使用研究を項目数、言語、対象、年で絞り込む
- 原版と選択中の後続版の差分を明示する
- 「短いが内容範囲が狭い」「翻訳済みだが妥当性未検証」などの注意を表示する
- 採用根拠文献をBibTeXまたはRISで出力する
- 同一概念内の尺度候補を比較しやすくする

単純な総合点やランキングを付ける場合は、評価基準と重みを公開する。根拠の異なる尺度をブラックボックスの点数だけで順位付けしない。

### P2: 尺度項目と利用条件

- 掲載可能な尺度だけ、項目本文・回答選択肢・逆転項目・得点化を登録
- 利用条件、出典、ライセンス、確認日を表示
- 掲載不可・不明の場合は、公式入手先への導線を改善

### P2: 派生関係の可視化

- 原版 → 短縮版
- 原版 → 翻訳版
- 原版 → 文脈調整版
- 類似概念・代替尺度

最初はテキストと表で実装し、データ構造が安定してから図にする。

### P2: 変更履歴と運用

- 尺度単位の`verifiedAt`と確認者
- データ変更履歴
- 根拠追加・訂正のIssueテンプレート
- 出典切れリンクの定期確認
- データ版番号とサイト機能版番号の整理

## 6. 次のモデルが最初に行う推奨タスク

大きな機能追加から始めず、次の順が安全である。

1. `node verify-data.mjs`で既存データを確認する。
2. Consumer Perceived Value / PERVALについて、短縮版の追加使用文脈と日本語根拠を拡充する。
3. 日本語版の開発・検証、または翻訳・使用根拠を1件確認する。
4. 原版・短縮版・研究内削除版の区別を明記する。
5. 構文・データ検証後にコミット・公開する。

一度に多数尺度を追加するより、1～3尺度ずつ根拠を確認して公開する方が誤登録を発見しやすい。

## 7. 作業開始・検証・公開手順

### 作業開始

`C:\myproject\AGENTS.md`を読み、対象フォルダで必ず実行する。

```powershell
git pull
git status --short
git diff --stat
```

```powershell
node --check app.js

node verify-data.mjs
git diff --check
```

データ検証スクリプト追加後は、必ずそれも実行する。

### 公開

```powershell
git add <変更したファイル>
git commit -m "変更内容を表す短い英語メッセージ"
git push
```

- `git push`: 保存した履歴をGitHubへ送る操作

その後、GitHub Pagesの`pages-build-deployment`成功を確認する。HTTP 200だけでなく、公開された`data.js`や`app.js`に新しい版番号・尺度名・機能名が含まれることをコマンドで確認する。

## 8. このスレッド固有の運用制約

ブラウザ起動・ブラウザ操作がCodex再起動の一因である可能性があるため、このスレッドではブラウザを開かない。

使用する検証手段:

- `node --check`
- データ検証スクリプト
- PowerShellの`Invoke-WebRequest`
- `gh run list` / `gh run watch`
- 公開中のJS・CSS内のマーカー確認

画面クリック、保存復元、レスポンシブ表示など、ブラウザが必要な検証は「未実施」と明記する。実施していない画面テストを成功と報告しない。

## 9. 既知の注意点

- 静的サイトなので、データ量増加に伴い`data.js`が大きくなる。
- `data.js`はJavaScriptオブジェクトであり、同じオブジェクト内のキー重複が構文エラーにならず、後の値で上書きされる。自動検知が必要。
- `verify-data.mjs`は管理下の`data.js`を検証するローカル用ツールであり、JavaScript実行を安全に隔離する仕組みではない。取得元を確認していない変更には実行しない。
- ローカル保存はブラウザの`localStorage`依存で、端末間同期はない。
- 利用研究53件は全尺度を均等にカバーしていない。
- 利用研究数が未集計の尺度が多い。
- 日本語情報未確認が24尺度ある。
- 尺度項目本文は未掲載。
- 公開ページの完全なクリック操作テストは、直近のCodex再起動により未完了。構文、データ整合性、GitHub Pages、公開ファイルのマーカーは確認済み。

## 10. 完了条件

各更新は、次を満たして完了とする。

1. 根拠資料から項目数・対象・回答形式・調整方法を確認した。
2. 原版、短縮版、翻訳版、研究内削除版を区別した。
3. 引用数を尺度使用数として登録していない。
4. 日本語根拠の段階を正しく分類した。
5. 構文とデータ検証が通った。
6. 関係ない既存変更を上書きしていない。
